# Предложения по развитию системы

Документ для решения: что делать дальше с прототипом, чтобы превратить его в **рабочий инструмент с непрерывным контролем качества**.

Главный запрос: **загружать данные по расписанию и сравнивать прогноз с фактом** для расчёта реального разбега точности.

---

## Блок 1. Журнал прогнозов и live-точность ⭐ (главный приоритет)

### 1.1. Forecast Journal — что сохранять

При каждом запуске `update_data.py` дополнительно сохранять **снапшот всего прогноза**:

| Поле | Описание |
|---|---|
| `run_id` | Уникальный ID запуска (datetime + хеш) |
| `prediction_date` | Дата, когда сделан прогноз (сегодня) |
| `as_of_close` | Цена меди на момент прогноза (P0) |
| `model` | XGBoost / ARIMA / GBM / MLP / Ensemble |
| `horizon_days` | 3 / 10 / 21 / 63 / 126 |
| `target_date` | Дата, на которую делается прогноз (P0_date + H) |
| `point` | Точечный прогноз |
| `p10`, `p25`, `p75`, `p90` | Квантильный коридор |
| `prob_up` | P(↑) |
| `regime` | Текущий режим Markov (calm/turbulent) |
| `regime_calm_prob` | Вероятность calm |
| `weights_json` | Веса ансамбля на момент прогноза |
| `verdict` | Сокращённый совет (buy/wait/watch/urgent) |

**Технически:** SQLite-файл `data/forecast_journal.db` или CSV-таблица `data/journal/forecasts_YYYY.csv`. Для MVP CSV проще.

**Объём:** 5 моделей × 5 горизонтов = 25 строк в день. За год = ~9 000 строк (мизер).

### 1.2. Actuals Reconciliation — сравнение с фактом

Раз в день: для каждой строки журнала, **у которой target_date ≤ сегодня** и ещё не заполнен `actual_price`, подтянуть фактическую цену из yfinance и записать:

| Поле | Расчёт |
|---|---|
| `actual_price` | Фактическая цена на target_date |
| `error_pct` | (point − actual) / actual × 100 |
| `abs_error_pct` | |error_pct| |
| `in_p10_p90` | 1 если p10 ≤ actual ≤ p90, иначе 0 |
| `direction_hit` | 1 если sign(point − P0) = sign(actual − P0), иначе 0 |
| `reconciled_at` | Когда записан факт |

### 1.3. Live-метрики (накопительно)

На основе reconciled-строк считать **скользящие** метрики за окна 30 / 90 / 365 дней:

- **Live-MAPE** — средняя процентная ошибка на реальных прогнозах
- **Live-Coverage80** — % попаданий факта в коридор p10-p90
- **Live-HitRate** — % угадывания направления

Сравнивать с **back-test метриками**:

| Метрика | Back-test | Live (90д) | Разница |
|---|---|---|---|
| MAPE 1 мес | 5.6% | ? | если live > back-test × 1.5 → деградация |
| Coverage80 1 мес | 80% | ? | если < 70% → коридор узкий |
| HitRate 1 мес | 60% | ? | если < 50% → модель шумит |

### 1.4. Новая вкладка «📈 Live-точность»

В Расширенном режиме:
- График **error vs time** для каждого горизонта (точки прогнозов, цветом — попал ли в коридор)
- График **накопленных метрик** Live-MAPE / Coverage80 / HitRate во времени
- Таблица «Прогноз vs Факт» за последние N дней
- Алерт, если live-метрики деградируют относительно back-test

**Объём работы:** ~2-3 дня разработки.

## Блок 2. Автоматизация по расписанию

### 2.1. Cron на VPS (рекомендую)

**Утренний цикл (08:30 МСК):**
```bash
# 1. Свежие данные
python update_data.py

# 2. Прогноз + сохранение в журнал
python journal_save.py

# 3. Подтянуть факты для прошлых прогнозов
python journal_reconcile.py
```

При наличии VPS (см. SUBSCRIPTIONS.md, ~400-1000 ₽/мес) — один cron-файл. **Получаем:**
- Каждое утро свежие данные и сохранённый прогноз
- К 9:00 в дашборде уже актуальная картина
- Журнал растёт сам, без участия человека

### 2.2. Альтернатива — GitHub Actions (бесплатно)

Если без VPS — workflow `.github/workflows/daily.yml`:
```yaml
on:
  schedule:
    - cron: '30 5 * * 1-5'   # 08:30 МСК в будни (UTC+3)
jobs:
  refresh:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run journal_save
        run: python journal_save.py
      - name: Commit journal
        run: git commit -am "Daily journal $(date)" && git push
```
GitHub коммитит CSV в репо → Streamlit Cloud видит свежий файл.

**Минусы:** журнал хранится в git (не идеально для больших объёмов), нет защиты от падений.

### 2.3. Уведомления (Telegram)

После каждого `journal_save.py` — отправка в Telegram-канал:
```
🟢 CopperCast · 2026-05-29 09:00
Спот: 14 063 USD/т (+0.4%)
1 мес: ↓ Подождать (12 690, −9.7%)
6 мес: ↑ Покупать (14 350, +1.7%)

Live-точность 1м (90д): MAPE 6.1%, Coverage80 76%
Алерт: Coverage80 ниже целевых 80%
```
Через `python-telegram-bot` за 1 день работы.

## Блок 3. Инфраструктура

### 3.1. БД вместо CSV (если журнал вырастет)

SQLite — компактно, не требует сервера:
```
data/coppercast.db
├── forecasts (run_id, model, horizon, point, p10, p90, ...)
├── actuals (target_date, actual_price)
├── reconciliations (run_id, model, horizon, error_pct, in_p10_p90, ...)
└── live_metrics (date, horizon, mape_90d, coverage_90d, hitrate_90d)
```

Преимущества: запросы быстрее, индексы по дате/модели, не сломается при росте.

### 3.2. Версионирование моделей

Каждый раз при изменении гиперпараметров — записывать `model_version` в журнал. Это позволит:
- Сравнить «было / стало» при изменении модели
- Отказаться от изменений, если live-метрики ухудшились
- A/B-тестирование: для одного дня запускать 2 версии модели в журнал

### 3.3. Backup журнала

Журнал — это **активная память системы**, его потеря критична. Решения:
- Ежедневный бекап CSV/SQLite на S3 / Selectel Object Storage (~50 ₽/мес)
- Или просто `git commit` журнала в репо (бесплатно, но не для больших объёмов)

## Блок 4. Адаптивная модель

### 4.1. Auto-retrain при деградации

Если `live_MAPE_30d > back-test_MAPE × 1.5` — триггерится переобучение:
- Расширяется окно тренировки (включая последние данные)
- Переподбираются веса ансамбля по live-точности (например, если XGB вдруг стал хуже на 1м — снижаем его вес автоматически)

### 4.2. Учёт фактической волатильности

GARCH(1,1) на live-residuals — обновлять оценку σ из реальных ошибок прогноза, а не только из исторических доходностей.

### 4.3. Conformal Prediction (калибровка коридоров)

Уже реализовано в Пакете 5, но **на back-test**. Перенастроить на **live-residuals** — гарантирует, что Coverage80 действительно = 80%, а не «случайно» 70-90%.

## Блок 5. Расширения UX

### 5.1. Журнал решений закупщика

В Сокращённом режиме — личный журнал:
- Что советовала система
- Что вы сделали (купили / подождали)
- Что произошло на самом деле
- «Точность совета за 90 дней: 4 из 5 = 80%»

### 5.2. Триггеры и алерты

Из прототипа CopperCast:
- «Уведомить, если цена опустится ниже X»
- «Если вердикт сменится на Срочно покупать»
- «Если выйдет supply_shock-новость»

Требует БД и Telegram-бота → синергия с Блоком 2.3.

### 5.3. Прогноз в рублях

Дополнительная модель USD/RUB (отдельный VAR или просто текущий курс ЦБ) → перевод USD/т в ₽/т для **бюджетных решений**. Закупщику это удобнее.

### 5.4. Анализ конкретного контракта

«Что если законтрактовать на 90 дней при цене 13 200?»
- Сравнение фиксированной формулы с прогнозом
- Симуляция прибыли/убытка при разных сценариях

## Блок 6. Развитие данных

### 6.1. Долгосрочное накопление LME 3M

При ежедневном `update_data.py` Westmetall дотягивает LME. Через **6 месяцев** будет ~150 дневных точек → можно сделать **отдельную LME-модель** (не только COMEX).

### 6.2. SHFE warehouse stocks (бесплатно)

Парсинг публичной страницы SHFE. Китайские склады — ключевой индикатор спроса. ~2 дня работы.

### 6.3. Yangshan Premium (бесплатно — чтение, платно — CSV)

Через SMM. Премия китайской таможни — самый быстрый индикатор китайского физического спроса.

### 6.4. ICSG / Platts / SMM (платное)

Из SUBSCRIPTIONS.md — фундаментальные данные баланса меди. Эффект +5-15% точности на длинных горизонтах.

## Рекомендуемый план: 3 фазы

### Фаза 1 (1-2 недели) — Journal + Live ⭐
1. `journal_save.py` — сохранение всех прогнозов в CSV/SQLite (день 1)
2. `journal_reconcile.py` — подтягивание факта когда target_date наступила (день 2)
3. Вкладка «📈 Live-точность» с графиками и метриками (день 3)
4. Cron на VPS / GitHub Actions для ежедневного запуска (день 4)
5. Telegram-уведомления при критических изменениях (день 5)

**Результат:** через 2-3 месяца работы — реальные метрики точности, а не только синтетический back-test.

### Фаза 2 (1 месяц) — Адаптация и UX
1. Auto-retrain при деградации live-метрик
2. Журнал решений закупщика (что советовали vs что произошло)
3. Триггеры и алерты
4. Прогноз в рублях через курс ЦБ
5. Conformal prediction на live-residuals

### Фаза 3 (2-3 месяца) — Расширение данных
1. SHFE stocks (бесплатно, парсинг)
2. Yangshan premium (если работаете с китайскими контрактами)
3. Платные источники (см. SUBSCRIPTIONS.md): ICSG, Platts, SMM
4. Через 6 мес накопления — отдельная LME-модель

## Что делать первым

**Конкретно по вашему запросу — фаза 1, пункты 1-3.**

Если скажете «делай» — реализую за 2-3 итерации:

1. **Журнал прогнозов** (`journal_save.py` + CSV) — сохранение каждого прогноза.
2. **Reconciliation** (`journal_reconcile.py`) — подтягивание факта.
3. **Вкладка «📈 Live-точность»** — графики реальной точности.

Без расписания это уже даёт ценность: каждый раз, когда вы открываете дашборд, в журнал записывается snapshot. Через месяц-два будет первая статистика.

С расписанием (cron на VPS) — система работает **сама**, журнал растёт автоматически.

Скажите, какие пункты делать в первую очередь и в каком порядке.
